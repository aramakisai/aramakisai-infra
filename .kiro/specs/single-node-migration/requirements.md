# 要件定義 (Requirements) - シングルノード移行

## 1. インフラ縮小

**1.1** システムは、3×CX23（4GB）のHAクラスターを1×CX33（8GB）のシングルノードに移行しなければならない。  
**1.2** 移行後のノード名は `prod-node-1` を維持し、DNSおよびgitopsマニフェストへの影響を最小化しなければならない。  
**1.3** 移行後の年間インフラコストは現状（¥32,200/年）から削減されなければならない。

## 2. データ移行

**2.1** Authentik PostgreSQL のすべてのデータ（ユーザー、トークン、設定）は移行先で利用可能でなければならない。  
**2.2** Directus PostgreSQL のすべてのデータ（コンテンツ、アセット管理情報）は移行先で利用可能でなければならない。  
**2.3** Roundcube SQLite のデータ（セッション、アドレス帳）は移行先で利用可能でなければならない。  
**2.4** Stalwart メールデータは移行時に破棄し、新規スタートとする（メールデータ移行は対象外）。

## 3. GitOps 適応

**3.1** 移行後、CNPGのDBクラスターはシングルインスタンス（Primary のみ）で動作しなければならない。  
**3.2** Stalwart StatefulSet の `nodeSelector` は削除され、ノード名に依存しない構成にしなければならない。  
**3.3** Hetzner S3認証情報はExternalSecretで注入され、マニフェストに平文で書いてはならない。  
**3.4** CNPGはバックアップスペック（`backup`）で設定されたS3バケットにWALおよびベースバックアップを継続的にアーカイブしなければならない（`backup`スペックが前提条件）。

## 4. 半自動コールドスタンバイ復旧

**4.1** Grafana Cloud Synthetic Monitoring は主要サービスエンドポイント（`idp.aramakisai.com`）を定期的に監視しなければならない（`monitoring`スペックが前提条件）。  
**4.2** ノード障害を検知した際、Grafana Cloud は Raspberry Pi 上の webhook エンドポイントに通知を送信しなければならない。  
**4.3** Raspberry Pi の復旧スクリプトは、通知受信から `terraform apply` および `ansible-playbook` を自動実行しなければならない。  
**4.4** 二重実行を防止するロック機構を持ち、復旧スクリプトが同時に複数起動しないようにしなければならない。  
**4.5** 復旧プロセス全体（通知受信からサービス復旧まで）は30分以内に完了しなければならない。

## 5. 移行後検証

**5.1** 移行後、すべての主要サービス（Authentik、Directus、Stalwart、Roundcube、ArgoCD）が正常動作することを確認しなければならない。  
**5.2** 移行後、CNPGのS3バックアップが動作していることを確認しなければならない。  
**5.3** 移行後、Raspberry Pi からクラスターへの疎通（Tailscale経由SSH）が確認されなければならない。
