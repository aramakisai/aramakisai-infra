# Design Document - ドキュメント更新フロー (docs-update-flow)

本設計書は、インフラ構成の変更時にプロジェクトの主要ドキュメント（プロジェクトメモリ）と実際のコードとの乖離（Staleドキュメント問題）を防ぎ、AIエージェントと開発者の両方が正確なコンテキストを利用できるようにするための「ドキュメント更新フロー」の設計を定義します。

---

## 1. 全体アーキテクチャ

本仕様で導入する更新フローと自動検証の全体像は以下の通りです。

```mermaid
graph TD
    A[開発者/AIエージェント] -->|コード/インフラの変更| B{変更の対象は?}
    
    B -->|変数/技術選定/監視等の追加| C[steering/tech.md の更新]
    B -->|ディレクトリ構造/サービス追加| D[steering/structure.md の更新]
    B -->|コマンド/手順等の追加| E[CLAUDE.md の更新]
    
    A -->|仕様完了時/PR作成時| F[プロジェクトメモリ同期プロセス]
    
    A -->|ローカル検証| G[scripts/check-links.py]
    G -->|Markdown内の相対パス存在チェック| H{不整合の有無}
    H -->|あり (リンク切れ)| I[エラー表示 / 修正]
    H -->|なし| J[検証成功]
```

## 2. コンポーネント設計

### 2.1. 主要ドキュメントの再構成 (REQ-01)

ドキュメントの役割分担を整理し、重複を排除しつつ、`CLAUDE.md` を本来の「日常的な運用手順書」として維持します。

#### 1. `CLAUDE.md` のリファクタリング
- **変更内容**:
  - 末尾の `Agentic SDLC and Spec-Driven Development` セクションを削除し、[GEMINI.md](../../GEMINI.md) へのリンク参照に一本化します。
  - テクニカルな詳細や意思決定（Ansibleの変数定義や設定フラグなど）は、原則として [steering/tech.md](../../.kiro/steering/tech.md) や [steering/structure.md](../../.kiro/steering/structure.md) に移譲します。
- **維持する内容**:
  - 日常的に開発者が使用するコマンド（Terraform, Ansible, ArgoCD操作等）。
  - ブートストラップフロー。
  - クラスターデータ・ディレクトリ等の基本的な構成。

#### 2. `.kiro/steering/tech.md` および `.kiro/steering/structure.md` の更新
- `CLAUDE.md` から移譲された、プロバイダーの詳細、変数一覧、詳細な設定フラグの解説、監視スタックの技術的な意思決定などを統合します。

### 2.2. プロジェクトメモリ（コンテキスト）のダイエット設計 (REQ-05)

エージェントへのシステムプロンプトやコンテキストウィンドウの肥大化による推論オーバーヘッドを解消するため、ドキュメントの冗長な記述を排除し、圧縮・整理する方針を定義します。

#### 1. 各ドキュメントのサイズ制限（上限ライン）
ドキュメントが継続的に肥大化することを防ぐため、以下の最大ライン数（目安）を設定します。
- `CLAUDE.md`: 最大 300 行（現状 715 行から 50% 以上削減）。
- `.kiro/steering/` 配下の各ドキュメント: 最大 150 行。

#### 2. ダイエットに向けた情報削除・圧縮方針
1. **一時的な情報/開発状況の除外**: 「既知のギャップ」や「将来的な構成案」「一時的な未完了ステータス」など、時間が経てば変わる開発途中の課題情報はドキュメント（特に `CLAUDE.md`）から除外し、GitHub Issue や PR ログに委ねる。
2. **構成管理手順の圧縮**: Ansible や Terraform の詳細なインストールコード（`tasks/main.yml` などからの一部転記）は削除し、リポジトリ内のコードそのものを Single Source of Truth として参照させる。
3. **記述の簡潔化**: 散文調の長い説明を廃止し、箇条書き（Bullet points）やキーと説明のシンプルな対応表に書き換える。

#### 3. ダイエットの実施タイミング
- Kiro の仕様完了（`phase: completed`）およびPRの作成時に、ドキュメントの更新と同時に「不要になった説明や冗長な記述の整理（リファクタリング）」をPRテンプレートのチェック項目として義務付ける。

### 2.3. インフラ変更時のドキュメント更新フローとAIエージェントルールの策定 (REQ-04)

インフラ構成や変数の変更などが発生した際、関連ドキュメントが漏れなく同期されるフローを規定し、AIエージェントが自律的にドキュメントを更新するためのルールを定義します。

#### 1. 変更時更新ナビゲーションフロー (`CLAUDE.md` への追加)
開発者やエージェントがインフラに変更を加えた際、どのドキュメントを更新すべきか判断するための以下のナビゲーション（チェックリスト）を `CLAUDE.md` に追加します。
- **Ansible タスク/ロールの変更**: `CLAUDE.md` の「コマンド」や、`steering/tech.md` の「Key Technical Decisions」に影響がないか確認し、更新する。
- **新規サービス/マニフェストの追加**: `steering/structure.md` の「実際の apps/prod/ 一覧」および `CLAUDE.md` の「ディレクトリ構成」に追記する。
- **Infisical 環境変数/シークレットの追加**: `CLAUDE.md` および `steering/tech.md` の「シークレット一覧」にキー名を追記する。

#### 2. AIエージェント向け Steering ルールの明文化
エージェントの行動指針となるよう、`.kiro/steering/structure.md` または `tech.md` 内に、以下の「ドキュメント同期ルール」を追加します。
> **[RULE] ドキュメントの自律的同期**:
> コード（Terraform、Ansible、GitOpsマニフェスト）に変更を加えた場合、エージェントは自律的に関連するドキュメント（`CLAUDE.md`、`steering/` 内ファイル）をスキャンし、最新の状態に同期しなければならない。コードの変更のみでタスクを完了してはならない。

### 2.4. Kiro仕様完了時のプロジェクトメモリ同期プロセス (REQ-02)

仕様完了時（`phase: completed` に更新するPRを作成する際）、インフラ変更によって生じた情報をプロジェクトメモリに同期するためのプロセスを定義します。

#### 1. 同期すべき情報の基準
以下の変更が発生した場合、プロジェクト主要ドキュメントへの転記・同期を必須とします。
1. **新規公開サービス・サブドメイン**: `CLAUDE.md` のサービス一覧、[steering/structure.md](../../.kiro/steering/structure.md) 等の追加。
2. **新規シークレット・環境変数キー**: `CLAUDE.md` の「変数・シークレット管理」セクション、[steering/tech.md](../../.kiro/steering/tech.md) 等の追加（※機密情報そのものは含めない）。
3. **コマンドや手順の変更**: 運用コマンドやブートストラップフローが変更された場合、`CLAUDE.md` の更新。

#### 2. エージェント用ルールの明文化
- [.kiro/steering/structure.md](../../.kiro/steering/structure.md) または [.kiro/steering/tech.md](../../.kiro/steering/tech.md) の末尾に「仕様完了時のドキュメント同期手順」を明記します。これにより、次回以降AIエージェントが自律的に同期処理を考慮するようにガイドします。

### 2.5. 相対リンク検証スクリプトの設計 (REQ-03)

Markdownファイル内に記述された「プロジェクト内部への相対リンク」が有効であることを保証するスクリプトをPythonで作成します。

#### 1. スクリプト仕様 (`scripts/check-links.py`)
- **パッケージ管理**: プロジェクトルールに従い、`uv` を用いて動作可能なスクリプトとして実装します。サードパーティ製パーサーに依存せず、標準ライブラリ（`re`, `pathlib`, `urllib` など）で構成します。
- **検証ロジック**:
  1. リポジトリ内のすべての Markdown ファイル（`*.md`）を再帰的に走査。
  2. 各ファイルの本文から、Markdown形式のリンク `[テキスト](パス)` を抽出。
  3. **対象外リンクのフィルタリング**:
     - `http://` や `https://` から始まる外部リンクは無視。
     - `#section` のような同一ファイル内のアンカーリンクは無視。
  4. **相対パスの解決**:
     - `file:///` から始まるリンク（例: `[GEMINI.md](file:///home/username/projects/aramakisai-infra/GEMINI.md)`）については、`/home/username/projects/aramakisai-infra/` などの環境依存の絶対パスプレフィックスを除去するか、リポジトリルートからのパスに変換して解決。
     - 通常の相対パス（例: `[README](../README.md)`）は、検証対象の Markdown ファイルがあるディレクトリを基準にパスを解決。
  5. **存在チェック**:
     - 解決されたパスが、実際にファイルまたはディレクトリとしてリポジトリ内に存在するかを確認。
     - リンク切れを検出した場合は標準エラー出力に詳細（ファイル名、行番号、無効なパス）を出力。
- **終了コード**: リンク切れが1つでもあれば `1`（失敗）、すべて有効であれば `0`（成功）。
